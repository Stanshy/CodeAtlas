# 產品診斷報告 — Sprint 12

**專案**: CodeAtlas
**診斷日期**: 2026-04-01
**診斷人**: PM
**背景**: Sprint 11 交付了三種故事視角功能，但老闆看實際畫面後認為「三個視角看起來幾乎沒差異，使用者看不懂」。問題根因：只改了佈局和顏色，沒改「看什麼」。Sprint 12 要做兩件大事：① 視覺風格從深色霓虹改為白紙黑格 ② 三種視角的呈現邏輯徹底重做（目錄鳥瞰、聚焦呼叫鏈、播放模式）。老闆已核准新版圖稿。

---

## Q1: 問題存在嗎？（Problem Validation）

| 檢查項 | 說明 |
|--------|------|
| 痛點來源 | 老闆直接看畫面後指出：① 系統框架 dagre 排成一條水平線，完全看不懂 ② 三個視角只是換顏色，看起來幾乎一樣 ③ 節點太多太散，沒有故事引導感。核心目標是「5 分鐘看懂專案」，但使用者切換視角時反應是「這三個有什麼不同？」 |
| 頻率 | 每次使用。這是產品核心體驗 |
| 嚴重度 | 極高 — 三種故事視角是產品的核心差異化功能，如果使用者看不懂差異，等於核心功能失效。同時深色主題在節點多時視覺混亂，白底更乾淨專業 |
| 現有替代方案 | 無。Sprint 11 已交付但效果不達標 |

**結論**: [x] 問題真實存在且值得解決

---

## Q2: 目標用戶是誰？（User Definition）

| 檢查項 | 說明 |
|--------|------|
| 用戶角色 | 所有使用者。視覺風格和呈現邏輯影響每一個人的第一印象 |
| 使用頻率 | 每次使用 |
| 技術水平 | 零門檻 — 白底更友善，目錄卡片比檔案節點更易懂 |
| 關鍵場景 | ① 打開 CodeAtlas → 系統框架 → 看到 10 個目錄卡片分三層 → 「原來這專案是這樣分的」 ② 切換邏輯運作 → 點一個入口 → 只有呼叫鏈亮起，其他全消失 → 「喔，登入要經過這些」 ③ 切換資料旅程 → 選入口 → 節點一個一個亮起 → 「資料從這裡到那裡」 |

**結論**: [x] 用戶輪廓清晰

---

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | **視覺風格改白底**：全局 CSS 主題替換，複雜度中等但影響範圍大（所有元件都要調色）。**系統框架改目錄卡片**：需要 core 層新增目錄聚合邏輯，web 層新增目錄節點渲染，複雜度中等。**聚焦模式**：基於 Sprint 8 影響分析 + Sprint 11 BFS hover，只需調整顯示邏輯（隱藏而非淡化），複雜度低。**播放模式**：Sprint 11 已有 stagger animation 基礎，需強化為「逐步出現」而非「逐步高亮」，複雜度低。整體價值極高 — 徹底解決「看不懂」問題 |
| 有沒有更簡單的做法 | 風格改白底是老闆明確要求，無替代。三種視角的呈現邏輯改動建立在現有程式碼上，不是重寫 |
| 技術可行性 | ✅ CSS 主題替換是標準做法。目錄聚合需要 core 層新增函式但邏輯簡單（按 directory group nodes）。聚焦/播放模式基於現有功能強化 |
| 邊界情況 | ① 白底風格需要重新調整所有元件（NeonNode → 不再 Neon？命名問題）② 深色主題是否保留為可選？→ 先做白底，深色暫不保留 ③ 目錄聚合在扁平專案（無目錄結構）下可能只有 1-2 個目錄 → 回退為檔案視圖 ④ 3D 模式需要適配白底 |

**結論**: [x] 解法合理

## Q4: 範圍明確嗎？（Scope Clarity）

| 檢查項 | 說明 |
|--------|------|
| 必做（Must Have） | ① 白紙黑格視覺主題（全局 CSS 替換）② 系統框架改為目錄鳥瞰（core 目錄聚合 + web 目錄卡片渲染 + dagre 分層）③ 邏輯運作改為聚焦模式（點擊入口 → 呼叫鏈亮起 → 其餘 opacity 0.08）④ 資料旅程改為播放模式（逐步出現 350ms + E2E 面板同步 + 重播）⑤ 嚴格按照核准圖稿實作（`proposal/references/sprint12/three-perspectives-mockup.html`）⑥ 全面回歸測試 |
| 應做（Should Have） | 3D 模式白底適配、之前累積的 P1（AI 輔助分類 + AI 設定 UI + Onboarding + 3D Bloom） |
| 不做（Out of Scope） | 深色主題保留為可選（只做白底）、Sequence Diagram 自動生成、Zoom LOD、Tour 功能、多語言 |
| 驗收標準 | 三種視角切換時使用者反應「哇，完全不同」。系統框架只有目錄卡片（5~15個）。邏輯運作點擊後只看到呼叫鏈。資料旅程逐步動畫播放。白底乾淨專業 |
| 依賴 | Sprint 11 三種視角框架（✅ 已完成）、Sprint 10 智慧策展（✅ 已完成）、核准圖稿（✅ 已完成） |

**結論**: [x] 範圍清晰可執行

## Q5: 符合架構嗎？（Architecture Alignment）

| 檢查項 | 說明 |
|--------|------|
| 三層分離原則 | ⚠️ 主要在 web 層（主題 + 渲染邏輯），但系統框架的目錄聚合需要 **core 層新增**目錄分組函式。core 產出目錄級 Graph JSON → web 渲染目錄卡片。符合 core 輸出資料、web 消費資料的架構 |
| 資料流向 | core 新增 `aggregateByDirectory()` → Graph JSON 新增目錄節點 → web graph-adapter 根據視角選擇渲染目錄或檔案。資料流正確 |
| 命名規範 | ⚠️ 元件名稱 `NeonNode`、`NeonEdge` 在白底風格下不再合適。建議：先保留命名不改（只改樣式），避免大規模重命名風險。命名重構放後續 |
| 歷史教訓 | ① Sprint 10 教訓：useMemo 重構後變數引用遺漏 → 主題替換時也要注意所有引用 ② Sprint 11 教訓：dagre 排成一條線 → 目錄聚合後節點只有 5~15 個，dagre 更容易分層成功 ③ Sprint 11 Review 教訓：NeonEdge 缺 CSS class → 主題替換時要逐一確認所有 CSS class |

**結論**: [x] 符合架構（core 需小幅新增目錄聚合）

## Q6: 風險可控嗎？（Risk Assessment）

| 風險類型 | 評估 |
|---------|------|
| 時間風險 | 🟡 中等 — 全局主題替換影響範圍大（所有元件的顏色、邊框、陰影都要改），但圖稿已有完整 CSS 變數定義，是機械性替換而非設計探索 |
| 技術風險 | 🟡 中等 — core 新增目錄聚合邏輯需設計（如何聚合、目錄間依賴怎麼計算）。3D 模式白底適配可能有 Three.js 材質問題 |
| 回歸風險 | 🟠 中高 — 全局主題替換 + 呈現邏輯改動，影響幾乎所有 web 層元件。1092 tests 保護，但 visual regression 需人工確認 |
| 維護風險 | 🟢 低 — CSS 變數化後，未來主題切換更容易。目錄聚合是穩定功能 |

**結論**: [x] 風險可控（中高回歸風險，需嚴格 Review + 人工視覺確認）
