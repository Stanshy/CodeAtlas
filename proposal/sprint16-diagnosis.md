## Q2: 目標用戶是誰？（User Definition）

| 檢查項 | 說明 |
|--------|------|
| 用戶角色 | 所有 CodeAtlas 用戶 — 工程師、Vibe coder、PM/QA/老闆 |
| 使用頻率 | 每次使用 CodeAtlas 都會觸發（核心功能，非邊緣功能） |
| 技術水平 | 非技術用戶應能在網頁 UI 完成所有 AI 設定，不應依賴 CLI flag |
| 關鍵場景 | **場景 1**：工程師接手陌生專案 → 開 CodeAtlas → 選擇感興趣的目錄 → AI 摘要讓他 5 分鐘看懂架構。**場景 2**：PM 要了解技術架構 → 看到中文 AI 摘要 + 端點描述 → 不用問工程師就能理解系統。 |

**結論**: [x] 用戶輪廓清晰

> 核心大原則：**用 AI 分析讓使用者能快速了解某個專案的架構、原理、邏輯以及資料流**（老闆原話）。

---

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | 中等複雜度。需跨 core+cli+web 三層，但每層改動範圍明確。價值極高 — 直接決定 AI 功能能否被用戶感知。 |
| 有沒有更簡單的做法 | 最小方案：只換 Provider + Prompt 中文化，就能大幅提升成功率。但老闆明確要求「不要一次掃全專案」，需要加入使用者控制的分析範圍。 |
| 技術可行性 | ✅ 可行。管線架構（Sprint 15.1）已通，Provider 切換是 createProvider() 一行的事。前端設定連動後端需要新 API 端點，但 server.ts 已有完整的 AI 端點模式可參照。 |
| 邊界情況 | (1) 使用者選了目錄但 AI Provider 未設定 → 需 graceful 提示 (2) 分析範圍過大（>500 方法）→ 需 progress + 分批 (3) Claude CLI 在 Windows 的 spawn 問題（Sprint 15.1 已修部分，需驗證） |

**結論**: [x] 解法合理

> 老闆的核心要求：**使用者主動選擇才分析**，不自動掃全專案。這是 UX 設計問題，技術上可行。

## Q4: 範圍明確嗎？（Scope Clarity）

| 檢查項 | 說明 |
|--------|------|
| 必做（Must Have） | (1) Provider 品質提升（換 Claude API / Prompt 中文化）(2) 使用者控制的分析範圍（不自動掃全專案）(3) 前端設定連動後端（選 Provider → 後端實際切換）(4) AI 結果持久化（重啟不遺失）(5) AI 內容呈現優化（讓用戶看得到、看得懂）(6) Claude CLI Windows 回應解析修復 |
| 不做（Out of Scope） | 多語言支援（Python/Java）→ Sprint 18+、增量分析（只分析變更檔案）→ 後續、AI streaming 逐字顯示、新增 AI Provider |
| 驗收標準 | 用戶在網頁選 Provider → 選目錄/端點 → 點「分析」→ 看到中文 AI 摘要，三視角 AI 成功率 >90%（使用 Claude API 時） |
| 依賴 | Sprint 15.1 管線已通（✅），需要用戶有 Claude API key 或本地 Ollama |

**結論**: [x] 範圍清晰可執行

## Q5: 符合架構嗎？（Architecture Alignment）

| 檢查項 | 說明 |
|--------|------|
| 架構原則 | core 只輸出 Graph JSON、cli 負責 serve、web 讀取 API — **全部符合**。新增的 `/api/ai/configure` 和 `/api/ai/analyze` 端點在 cli 層，符合「cli 負責編排」原則。 |
| 資料流向 | 前端 Settings → POST /api/ai/configure → cli 寫 .codeatlas.json → pipeline 用新設定 → /api/graph 合併結果 → 前端顯示。**流向清晰，無繞路**。 |
| 命名規範 | 新 API 端點沿用 `/api/ai/*` 命名空間，camelCase JSON。**符合**。 |
| 歷史教訓 | postmortem 記錄「API 端點遺漏」教訓 → 本次須逐端點對照 api-design.md。Sprint 15.1 的 9 項 Hotfix（Claude CLI Windows、Zod null、Batch size）都需在本次確認已穩定。 |

**結論**: [x] 符合架構

## Q6: 風險可控嗎？（Risk Assessment）

| 風險類型 | 評估 |
|---------|------|
| 時間風險 | 🟡 中等 — 功能項多（6 個 P0），但每項改動不大。重點是 UX 體驗設計需要思考。 |
| 技術風險 | 🟢 低 — 管線已通，主要是 Provider 切換 + 前端 UX + 持久化，無新技術風險。 |
| 回歸風險 | 🟢 低 — 1707 tests 做護城河，AI 功能改動不影響 core 解析引擎。 |
| 維護風險 | 🟢 低 — .codeatlas.json 設定持久化是標準做法，不增加架構複雜度。 |

**最大風險**：UX 設計。「使用者選擇分析範圍」的互動流程如果設計不好，會讓用戶困惑（什麼時候該分析？分析哪裡？結果在哪看？）。需要 PM 和 TL 仔細思考體驗流程。

**結論**: [x] 風險可控
