## Q1: 問題存在嗎？（Problem Validation）

> Sprint 1 做出了解析引擎和 graph JSON，但使用者看到的只是佔位頁。真正的產品價值 — 視覺化地圖 — 還不存在。

| 檢查項 | 評估 |
|--------|------|
| 痛點來源 | **產品核心價值尚未兌現**。Sprint 1 只完成了「分析」，還沒完成「視覺化」。用戶裝了工具跑 analyze 後看到的是空白佔位頁，沒有任何可用的視覺產出。 |
| 頻率 | **阻斷性**。沒有視覺化，產品等於不存在。這不是「有多常碰到」的問題，是「核心功能完全缺失」。 |
| 嚴重度 | **最高**。視覺衝擊力是老闆定義的核心賣點。不做這步，後面所有 Sprint 都沒有意義。 |
| 現有替代方案 | Sprint 1 的 graph JSON 可以手動用其他工具渲染，但這完全不是產品體驗。 |

**結論**: [x] 問題真實存在且值得解決

> 這是產品核心功能，不做就沒有產品。

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 評估 |
|--------|------|
| 複雜度 vs 價值 | **高複雜度，高價值**。React Flow + D3 force + 自訂動畫 + 深色霓虹主題，前端工作量大。但這正是產品的核心差異化，值得投入。 |
| 有沒有更簡單的做法 | ① 用 Mermaid 直接渲染 → 醜、無互動、無動畫 ② 用 Cytoscape.js → 互動可以但視覺自訂度不如 React Flow + D3 ③ 純 D3 → 太底層，開發成本更高。**React Flow + D3 是最合理的組合。** |
| 技術可行性 | **可行**。React Flow 有大量生產案例，D3 force simulation 是成熟技術。Sprint 1 的 graph JSON 格式已經設計好，可直接對接 React Flow 的 nodes/edges 格式。 |
| 邊界情況 | ① 大專案（500+ 節點）的渲染效能 → Sprint 4 處理，Sprint 2 先跑中型專案 ② 節點重疊 → 力導向佈局自動避免，但密集依賴的模組可能聚團 → 需要調參 ③ 深色主題下文字可讀性 → 需要設計審核 |

**結論**: [x] 解法合理

> React Flow + D3 + Canvas 動畫是前端圖譜渲染的最合理技術選型。

## Q6: 風險可控嗎？（Risk Assessment）

| 風險類型 | 評估 |
|---------|------|
| 時間風險 | **中高**。8 項視覺功能全做，前端工作量大。力導向佈局的調參、動畫效果的打磨都容易超時。 |
| 技術風險 | **中**。React Flow + D3 整合有成熟方案，但自訂 Canvas 動畫（粒子流動）需要額外研究。 |
| 回歸風險 | **低**。純新增 web package，不改 core/cli。 |
| 維護風險 | **中**。動畫和視覺效果的程式碼容易變得難維護，需要好的元件拆分。 |

**最壞情況**：視覺效果全做完但效能太差，大專案卡頓。
**承受能力**：可以。Sprint 2 先確保中型專案（100-300 檔案）流暢，效能優化放 Sprint 4。

**結論**: [x] 中等風險需監控

> 時間風險是主要挑戰。建議 Sprint 2 內部拆分為：先做靜態截圖能成立的效果，再疊動畫。
