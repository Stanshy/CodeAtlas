## Q1: 問題存在嗎？（Problem Validation）

| 檢查項 | 說明 |
|--------|------|
| 痛點來源 | **老闆直覺**：「程式碼好像太多了」。16 個 Sprint 累積，疑似膨脹。 |
| 頻率 | 每次開發都感受到 — 改一個功能要動很多檔案、Context window 負擔大 |
| 嚴重度 | 中 — 不影響用戶，但影響開發效率和 AI Agent 的理解能力 |
| 現有替代方案 | 目前沒有替代方案，只能忍受 |

**實際分析**：
- 35K 產品碼對一個 monorepo（core+cli+web）的分析+視覺化工具來說**不算過大**
- 但 web 佔 70% 且有多個 600+ 行大元件，確實有拆分空間
- ControlPanel.tsx（609 行）可能已被 SettingsPopover（899 行）取代，疑似死碼
- GraphCanvas.tsx（2,644 行）和 Graph3DCanvas.tsx（1,627 行）是否有共用邏輯可抽取
- 測試碼 21,600 行（佔總量 38%），比例合理但可能有冗餘

**結論**: [x] 問題存在但優先級中等 — 不是「太多」而是「可以更精簡」

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | 低複雜度（重構不是新功能）、中等價值（開發效率+開源準備） |
| 更簡單的做法 | 可以只做「盤點+死碼清理」而不做深度重構，風險更低 |
| 技術可行性 | 完全可行，都是現有程式碼的整理 |
| 邊界情況 | 重構過程中可能引入回歸 bug，需要完善的測試覆蓋保護 |

**建議分層**：
1. **安全層**（低風險）：死碼移除、未使用 import 清理、debug log 清理
2. **中等層**（中風險）：大元件拆分（GraphCanvas、SettingsPopover）、共用邏輯抽取
3. **深度層**（高風險）：模組架構重組 — 建議不做或慎重評估

**結論**: [x] 解法合理，但建議以安全層+中等層為主，不做深度架構重組
