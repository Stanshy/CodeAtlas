## Q1: 問題存在嗎？（Problem Validation）

| 檢查項 | 說明 |
|--------|------|
| 痛點來源 | 老闆直接指出：經過 8 個 Sprint 功能累積，缺少統一控制面板、畫面呈現方式無法切換、輸入/輸出資料路徑不夠直覺。功能「藏」在右鍵、雙擊、搜尋欄等不同入口，使用者難以發現和使用 |
| 頻率 | 每次開啟工具都會面對 — 不知道有哪些功能可用、不知道怎麼設定顯示偏好 |
| 嚴重度 | 高 — 影響產品第一印象和使用深度。功能做了但用不到，等於白做。Sprint 8 的 FilterPanel 和 OverviewPanel 甚至連按鈕都未連接 |
| 現有替代方案 | 使用者需記住：右鍵→影響分析、雙擊→展開函式、搜尋欄→路徑追蹤。無文件指引、無設定 UI、無視圖切換。學習成本高 |

**結論**: [x] 問題真實存在且值得解決

## Q2: 目標用戶是誰？（User Definition）

| 檢查項 | 說明 |
|--------|------|
| 用戶角色 | ① 所有使用者（控制面板是基礎 UX）② 工程師（切換依賴/資料流/呼叫鏈視圖）③ 新手（Onboarding 提示引導）④ PM/非技術人員（簡化操作入口） |
| 使用頻率 | 控制面板：每次使用。視圖切換：每次探索不同面向。設定：首次 + 偶爾調整 |
| 技術水平 | 控制面板和視圖切換應該零門檻。「點一下就看到不同角度」 |
| 關鍵場景 | ① 打開工具 → 控制面板一目了然有哪些功能 → 點切換視圖 ② 切到「資料流視圖」→ 只看資料進出路徑 → 亮起整條路線 ③ 設定面板中切換 AI Provider → 不用回 CLI |

**結論**: [x] 用戶輪廓清晰

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | 控制面板：純 UI 組件，複雜度低，價值極高（統一所有入口）。視圖模式：基於現有 filter + highlight 邏輯組合預設配置，複雜度中。端到端資料流追蹤：整合 Sprint 5 路徑追蹤 + Sprint 7 呼叫鏈 + Sprint 8 影響分析，複雜度中。整體價值高 |
| 有沒有更簡單的做法 | 視圖模式可以做成「預設過濾 + 預設高亮」的組合，不需要全新渲染邏輯。控制面板可以復用現有 ViewStateContext 的 dispatch。端到端追蹤可以串接現有 BFS |
| 技術可行性 | ✅ 全部在 web 層完成，不改 core/cli。ViewStateContext 已有成熟的擴充模式（Sprint 5/7/8 各擴充過）。FilterPanel/OverviewPanel 已建好元件，只差 Toolbar wiring |
| 邊界情況 | ① 視圖模式切換時狀態重置（影響分析/搜尋聚焦要清除嗎？）② 端到端追蹤在大型專案可能路徑過長 → 復用 depth limit ③ 控制面板在小螢幕需要可收合 |

**結論**: [x] 解法合理

## Q4: 範圍明確嗎？（Scope Clarity）

| 檢查項 | 說明 |
|--------|------|
| 必做（Must Have） | ① 控制面板（Settings Panel）— 統一所有 toggle 和設定 ② 視圖模式切換 — 至少 3 種預設視圖（全景/依賴/資料流）③ 端到端資料流追蹤 — 選入口 → 追蹤 input 到 output → 路徑發亮 ④ 顯示偏好 — Heatmap/邊標籤/粒子動畫/節點標籤 toggle ⑤ Toolbar 修復 — FilterPanel + OverviewPanel 按鈕連接 ⑥ 2D + 3D 適配 |
| 應做（Should Have） | AI 設定 UI（Provider 切換 + 隱私模式）、快捷鍵提示（Onboarding overlay） |
| 不做（Out of Scope） | 自訂主題/色彩（Phase 5）、佈局持久化到 .codeatlas.json（未來考慮）、鍵盤快捷鍵自訂、拖拽面板重新排列 |
| 驗收標準 | 控制面板可存取所有功能 → 視圖一鍵切換 → 資料流路徑發亮 → 所有 toggle 立即生效 |
| 依賴 | Sprint 5 路徑追蹤 BFS、Sprint 7 呼叫鏈、Sprint 8 ViewStateContext 擴充（影響分析/過濾/搜尋聚焦） |

**結論**: [x] 範圍清晰可執行

## Q5: 符合架構嗎？（Architecture Alignment）

| 檢查項 | 說明 |
|--------|------|
| 三層分離原則 | ✅ 全部在 web 層完成。控制面板、視圖模式、端到端追蹤均為前端 UI + ViewState 邏輯。不改 core（解析引擎）、不改 cli（API 端點）。唯一例外：AI 設定 UI 如果要在 UI 切換 provider，可能需要 cli 新增一個 POST /api/settings 端點（P1，視範圍） |
| 資料流向 | ✅ 控制面板 → dispatch ViewState action → 渲染層讀取狀態 → 更新畫面。與 Sprint 5/7/8 的 ViewStateContext 模式完全一致 |
| 命名規範 | ✅ `ControlPanel.tsx`、`ViewModeSelector.tsx`、`useEndToEndTrace.ts`、`SettingsPanel.tsx` — 符合 kebab-case 檔名 + PascalCase 元件 |
| 歷史教訓 | ① postmortem：API 端點遺漏 → 本 Sprint 主要是前端，端點風險低 ② Sprint 8 教訓：過濾邏輯放 graph-adapter 而非渲染元件 → 視圖模式切換也應在 adapter 層處理 ③ ViewState action 全大寫 + 底線命名慣例需遵循 |

**結論**: [x] 符合架構
