## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | 中高複雜度但價值極高。core 需新增 API 端點識別，web 需改三種視角的資料源和渲染。但這是產品從「看檔案」到「看邏輯」的質變 |
| 有沒有更簡單的做法 | Sprint 7 已做了函式級解析（tree-sitter），core 已有方法/class 資料。問題是 web 沒用上。不需要重做解析，只需要讓 web 層使用函式級數據 |
| 技術可行性 | 可行。core 已有 FunctionNode/ClassNode 型別（Sprint 7）。API 端點識別需新增 pattern matching（Express/Fastify router 定義）。圖稿和技術規格書都已完成 |
| 邊界情況 | ① 非標準 API 框架（自定義 router）可能識別不到 → 回退為檔案級 ② 方法呼叫鏈太深（>10 層）→ 截斷 + 提示 ③ 扁平專案無明確分層 → 智慧聚合回退 |

**結論**: [x] 解法合理 — 建立在 Sprint 7 函式級解析基礎上，不是從零開始

## Q6: 風險可控嗎？（Risk Assessment）

| 風險類型 | 評估 |
|---------|------|
| 時間風險 | 中 — core 端點識別 + web 三視角改造工程量不小，但圖稿和規格書已備齊，減少探索時間 |
| 技術風險 | 中 — API 端點識別依賴 pattern matching，非標準框架可能漏判。回退機制（fallback 檔案級）可降低風險 |
| 回歸風險 | 中高 — 三種視角資料源從檔案級改為方法/端點級，影響範圍大。1235 tests 保護 |
| 維護風險 | 低 — 端點識別為獨立模組，不影響既有解析。未來擴展多語言時 pattern 需擴充 |

**結論**: [x] 中等風險需監控 — API 端點識別是新能力，需有 fallback；三視角改動大但有測試保護
