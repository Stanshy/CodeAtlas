# 產品診斷報告 — Sprint 11

**專案**: CodeAtlas
**診斷日期**: 2026-04-01
**診斷人**: PM
**背景**: Sprint 10 完成智慧策展 + 效能 + 3D 空間。Sprint 11 是老闆決策 #17「智慧策展 + 三種故事視角」的第二期 — 三種故事視角。老闆已核准設計總監的視覺方向：「三視角、三佈局、三色調」。

---

## Q1: 問題存在嗎？（Problem Validation）

| 檢查項 | 說明 |
|--------|------|
| 痛點來源 | 老闆決策 #17（2026-04-01）：核心目標是「5 分鐘看懂一個陌生專案」，但目前只有一種呈現方式（力導向圖），使用者看到一團節點不知從何理解。需要用「故事」引導使用者：先看框架（這系統長什麼樣）→ 再看邏輯（功能怎麼運作）→ 再看資料（一筆資料怎麼走）。Sprint 9 的四種視圖模式（全景/依賴/資料流/呼叫鏈）是技術維度切分，不是使用者理解維度 |
| 頻率 | 每次使用。新手第一次打開 CodeAtlas 就需要這三種視角 |
| 嚴重度 | 高 — 這是產品核心體驗的最後一哩路。Sprint 10 解決了「看得清楚」（策展），Sprint 11 要解決「看得懂」（故事化）。沒有故事視角，使用者只能自行探索，無法 5 分鐘看懂 |
| 現有替代方案 | Sprint 9 的四種視圖模式提供了基礎切換，但缺乏引導性。使用者需要自行理解每種模式的差異 |

**結論**: [x] 問題真實存在且值得解決

## Q2: 目標用戶是誰？（User Definition）

| 檢查項 | 說明 |
|--------|------|
| 用戶角色 | 所有使用者，特別是：① 工程師第一次接手陌生專案 ② PM/QA/老闆需要理解系統架構 ③ Vibe coder 想快速看懂專案全貌 |
| 使用頻率 | 每次使用 CodeAtlas 時選擇視角 |
| 技術水平 | 零門檻 — 三種視角用人類語言命名（系統框架/邏輯運作/資料旅程），不需理解技術概念 |
| 關鍵場景 | ① 新人打開 CodeAtlas → 選「系統框架」→ 分層圖一目了然 → 「原來這專案分三層」 ② 想了解登入功能 → 選「邏輯運作」→ hover 路由 → 呼叫鏈亮起 → 「喔，登入要經過這些模組」 ③ 想追蹤一筆訂單 → 選「資料旅程」→ 選入口 → 路徑逐步亮起 → 「資料從 API 到 Service 到 DB」 |

**結論**: [x] 用戶輪廓清晰

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | **系統框架**：dagre 分層佈局，React Flow 原生支援，複雜度低，價值極高（30 秒看懂架構）。**邏輯運作**：強化現有力導向 + BFS hover 高亮，複雜度低，在 Sprint 8 影響分析基礎上增強。**資料旅程**：基於 Sprint 9 端到端追蹤加入 stagger animation + Bloom，複雜度中等，視覺效果最炫 |
| 有沒有更簡單的做法 | 三種視角都建立在現有基礎上：dagre 已在 React Flow 生態、BFS hover 已在 Sprint 8 影響分析實作、E2E 追蹤已在 Sprint 9。設計總監評估額外開發量約 3-4 天，無需新依賴 |
| 技術可行性 | ✅ dagre（React Flow 內建）、framer-motion（已安裝）、three（已安裝）。三種佈局引擎切換需要在 graph-adapter 層做路由，架構上合理 |
| 邊界情況 | ① 切換視角時佈局重新計算可能閃爍 → 需過渡動畫 ② dagre 分層佈局在非典型專案結構下可能不理想 → 回退為力導向 ③ stagger animation 節點多時會太慢 → 設定上限（如 > 30 節點時加速或簡化）④ 三種視角取代四種視圖模式，需處理向下相容 |

**結論**: [x] 解法合理

## Q5: 符合架構嗎？（Architecture Alignment）

| 檢查項 | 說明 |
|--------|------|
| 三層分離原則 | ✅ 全部在 web 層。core 不改（role 分類已在 Sprint 10 完成）。cli 不改。佈局引擎切換、視角狀態、動畫都在 web 層 |
| 資料流向 | ✅ 不改資料流。core 產出含 role 的 Graph JSON → cli serve → web 讀取 → graph-adapter 根據視角切換佈局和過濾 → 渲染。新增的只是 graph-adapter 層的佈局路由邏輯 |
| 命名規範 | ✅ `story-perspective.ts`（視角定義）、`dagre-layout.ts`（分層佈局）、`stagger-animation.ts`（動畫）符合 kebab-case 慣例 |
| 歷史教訓 | ① Sprint 8 教訓：過濾邏輯放 graph-adapter 層 → 佈局引擎切換也應在此層 ② Sprint 9 教訓：ViewMode 切換已有框架，故事視角可復用/替換 ③ Sprint 10 教訓：T10 useMemo 重構後變數引用遺漏 → 佈局引擎切換時需確保所有消費端都正確引用 ④ postmortem：無直接相關踩坑 |

**結論**: [x] 符合架構

## Q6: 風險可控嗎？（Risk Assessment）

| 風險類型 | 評估 |
|---------|------|
| 時間風險 | 🟢 低 — 設計總監評估 3-4 天額外開發量，三種視角都建立在現有基礎上，無需新依賴。Sprint 10 P1 收尾可視時間納入 |

| 維護風險 | 🟢 低 — 三種視角是長期產品結構，不是臨時方案。佈局引擎切換機制可擴展 |

**結論**: [x] 風險可控（中等偏低，需注意 2D/3D 雙模式適配和視圖模式重構回歸）
