## Q1: 問題存在嗎？（Problem Validation）

| 檢查項 | 說明 |
|--------|------|
| 痛點來源 | 使用者已能看到模組依賴 + 資料流 + 函式呼叫鏈，但無法回答「改了這個函式，哪些地方會受影響？」和「這個模組依賴的上游是誰？」。此外搜尋只能精確匹配名稱，無法用自然語言描述意圖（「處理登入的邏輯」） |
| 頻率 | 每次修改程式碼前都想知道影響範圍。每次探索陌生專案都想用自然語言搜尋 |

| 現有替代方案 | 手動追蹤依賴路徑（Sprint 5 路徑追蹤 + Sprint 7 呼叫鏈可做到，但需使用者自己一個個找） |

**結論**: [x] 問題真實存在且值得解決

## Q2: 目標用戶是誰？（User Definition）

| 檢查項 | 說明 |
|--------|------|
| 用戶角色 | ① 工程師（改 code 前預覽影響範圍）② Tech Lead（code review 時判斷改動風險）③ PM/非技術人員（自然語言搜尋理解專案）④ 新人（AI 專案概述快速上手） |
| 使用頻率 | 影響分析：每次改動前。搜尋：每次探索時。AI 概述：第一次接觸專案 |
| 技術水平 | 影響分析需理解依賴概念。自然語言搜尋 + AI 概述零門檻 |
| 關鍵場景 | ① 右鍵節點→「顯示影響範圍」→ 下游全部亮起 ② 輸入「處理登入的邏輯」→ AI 定位相關節點 ③ 一鍵「專案概述」→ 生成架構摘要 |

**結論**: [x] 用戶輪廓清晰

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | 影響分析：圖遍歷演算法（BFS/DFS），基於現有 edges 資料，複雜度中等。搜尋強化 + AI 概述：利用現有 AI Provider 介面，複雜度低。價值高 |
| 有沒有更簡單的做法 | 影響分析可以只做正向（下游），但反向（上游）同樣重要且實作成本差不多。自然語言搜尋可以先做簡單的 AI 關鍵字提取再匹配 |
| 技術可行性 | ✅ 影響分析：BFS/DFS on graph，Sprint 5/7 已有路徑追蹤基礎可復用。自然語言搜尋：AI 提取關鍵字 → 匹配節點 label/path。AI 概述：遍歷全圖 → 組裝 prompt → AI 生成摘要 |
| 邊界情況 | ① 大型專案影響範圍可能很大 → 需 depth limit + 視覺分層 ② AI 概述需要送出全圖結構（隱私考量 → 只送結構不送原始碼）③ 自然語言搜尋品質取決於 AI 模型 |

**結論**: [x] 解法合理

## Q4: 範圍明確嗎？（Scope Clarity）

| 必做（Must Have） | ① 正向影響分析（下游）② 反向依賴分析（上游）③ 搜尋聚焦模式（搜尋後圖暗掉，只有相關路徑亮）④ 模組過濾（按目錄/層級/類型）⑤ AI 專案概述（一鍵生成架構摘要）⑥ 2D + 3D 適配 |

| 驗收標準 | 選節點 → 下游/上游高亮 + 搜尋聚焦 + AI 概述可用 |
| 依賴 | Sprint 5 路徑追蹤 BFS、Sprint 7 函式級圖、Sprint 6 AI Provider 介面 |
