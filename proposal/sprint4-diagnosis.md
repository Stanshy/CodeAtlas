# 產品診斷報告 — Sprint 4（v2）

**專案**: CodeAtlas
**診斷日期**: 2026-03-31
**診斷人**: PM
**說明**: 路線圖 v2.0 核准後重寫。Sprint 4 目標從「整合打磨 + 開源上線」改為「3D 視覺化」。

---

## Q1: 問題存在嗎？（Problem Validation）

| 檢查項 | 說明 |
|--------|------|
| 痛點來源 | 目前只有 2D 平面圖，對於 100+ 檔案的專案，平面佈局擁擠、層級不明顯。3D 空間能天然展開複雜依賴，視覺衝擊力也是產品核心賣點（老闆決策 #3、#12） |
| 頻率 | 每次使用都會感受到 — 2D 大型專案節點重疊問題 |
| 嚴重度 | 高 — 老闆明確要求 3D，且「視覺衝擊力就是產品本身」 |
| 現有替代方案 | 無。React Flow 是 2D 框架，無法做 3D。需引入新渲染引擎 |

**結論**: [x] 問題真實存在且值得解決

---

## Q2: 目標用戶是誰？（User Definition）

| 檢查項 | 說明 |
|--------|------|
| 用戶角色 | 同 Phase 1：工程師、Vibe coder、非技術角色。但 3D 模式特別吸引「第一眼用戶」—— README 截圖/GIF 就讓人想試 |
| 使用頻率 | 3D 看全景、2D 看細節 — 互補使用 |
| 技術水平 | 不需額外安裝，WebGL 瀏覽器原生支援 |
| 關鍵場景 | ① 打開 3D 星圖 → 旋轉/飛入 → 「太酷了」② 大型專案在 3D 空間展開 → 層級清晰 ③ 3D hover → 相關路徑在空間中亮起 |

**結論**: [x] 用戶輪廓清晰

---

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | `3d-force-graph` 封裝了 Three.js + d3-force-3d，API 簡潔。引入成本中等，但視覺價值極高 |
| 有沒有更簡單的做法 | 有 — CSS 3D transform 假 3D，但效果差距大且不支援真正的 3D 互動。不考慮 |
| 技術可行性 | ✅ `3d-force-graph` 是成熟開源庫（GitHub 4k+ stars），支援節點自訂、力導向、hover 事件 |
| 邊界情況 | ① 2D↔3D 切換時的狀態同步（選中節點、搜尋結果）② 3D 下面板/搜尋的 DOM overlay 定位 ③ 行動裝置 WebGL 相容性（Phase 5 才管）|

**結論**: [x] 解法合理

---

## Q4: 範圍明確嗎？（Scope Clarity）

| 檢查項 | 說明 |
|--------|------|
| 必做（Must Have） | ① 3D 力導向渲染 ② 2D/3D 一鍵切換 ③ 3D 霓虹主題（延續色彩系統）④ 3D 互動（旋轉/飛入/點擊聚焦）⑤ 3D hover 高亮 ⑥ 3D 粒子流動 ⑦ 面板 + 搜尋 3D 適配 |
| 不做（Out of Scope） | 資料流動視覺化（Sprint 5）、Ollama（Sprint 6）、效能優化（Sprint 10）、行動裝置適配 |
| 驗收標準 | 2D/3D 可切換 + 3D 模式下旋轉/飛入/hover 高亮/粒子流動可用 + 面板搜尋正常運作 |
| 依賴 | Sprint 1-3 全部完成 ✅，web 包已有 React Flow 2D 渲染和面板/搜尋 |

**結論**: [x] 範圍清晰可執行

## Q5: 符合架構嗎？（Architecture Alignment）

| 檢查項 | 說明 |
|--------|------|
| 三層分離原則 | ✅ 不破壞。3D 渲染完全在 web 層，core 和 cli 不受影響 |
| 資料流向 | ✅ 不變。3D 渲染器和 2D 一樣讀 `/api/graph` 的 Graph JSON |
| 命名規範 | ✅ 新增檔案延續 kebab-case：`graph-3d.tsx`、`use-3d-controls.ts` |
| 歷史教訓 | ① 2D adapter 模式可參考 — 3D 同樣需要 adapter 層隔離 ② 狀態管理需統一 — 切換模式時共享選中/搜尋狀態 |

**結論**: [x] 符合架構
