## 1. 目標

將 Sprint 1 產出的 graph JSON 渲染成可互動的視覺化依賴圖，以深色霓虹風格 + 力導向佈局 + 流動動畫呈現，讓使用者打開 `codeatlas web` 的第一眼就覺得「這也太酷了」，截圖放到推特上就有傳播力。

這是產品核心價值的第一次兌現。Sprint 1 做了引擎，Sprint 2 讓它被看見。

## 5. 風險評估

| 風險 | 可能性 | 影響 | 緩解措施 |
|------|--------|------|---------|
| React Flow + D3 force 整合複雜度 | 中 | 高 | 先做 React Flow 原生佈局，再用 D3 force 替換 |
| 深色霓虹主題調色耗時 | 高 | 中 | UI designer 先出設計稿，確認後再實作，不邊做邊改 |
| Canvas 粒子動畫效能 | 中 | 中 | 先用 CSS animation 做簡化版，效果不夠再改 Canvas |
| 節點分層邏輯複雜 | 中 | 中 | 先做兩層（目錄/檔案），第三層（函式）放 Phase 2 |
| Sprint 1 API 回傳格式不適合前端 | 低 | 高 | 開工前先確認 graph JSON → React Flow 格式轉換可行 |

## 6. 失敗模式分析

| 失敗場景 | 可能性 | 影響 | 偵測方式 | 緩解措施 |
|---------|--------|------|---------|---------|
| 力導向佈局節點重疊嚴重 | 中 | 高 | 用真實專案測試 | D3 force 調參（charge/distance），加碰撞偵測 |
| 動畫太花導致使用者頭暈 | 低 | 中 | 內部測試反饋 | 動畫速度/透明度可調，或提供「減少動畫」模式 |
| 深色背景下文字不可讀 | 中 | 高 | 圖稿審核階段發現 | G1 圖稿審核必須檢查文字對比度 |
| Graph JSON 結構與 React Flow 不相容 | 低 | 高 | Sprint 2 開工第一天驗證 | 寫轉換層 adapter，不直接改 core 輸出格式 |

## 9. 驗收標準

### 核心視覺

- [ ] `codeatlas web` 在瀏覽器中顯示可互動的依賴圖（非佔位頁）
- [ ] 深色背景 + 霓虹色節點與邊線，視覺有衝擊力
- [ ] 力導向佈局：節點自動排列，不重疊，結構清晰

- [ ] 邊上有方向性流動動畫（光點/粒子）
- [ ] 展開/收合/聚焦有平滑過渡動畫

### 技術

- [ ] 中型專案（100-300 檔案）渲染流暢（> 30 FPS）

- [ ] web package 可獨立 build
