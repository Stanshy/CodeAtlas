# 產品診斷報告 — Sprint 10

**專案**: CodeAtlas
**診斷日期**: 2026-04-01
**診斷人**: PM
**背景**: 老闆重大方向調整 — 從「顯示所有檔案」轉為「智慧策展 + 三種故事視角」。Sprint 10 負責智慧策展 + 效能 + 3D 空間參考系。

---

## Q1: 問題存在嗎？（Problem Validation）

| 檢查項 | 說明 |
|--------|------|
| 痛點來源 | 老闆直接體驗後指出：① 畫面卡頓 — 9 個 Sprint 堆疊功能導致 re-render 過重 ② 3D 模式節點擠成一團 + 純黑背景無空間感 — 看不出「星圖」效果，無法判斷節點位置和深度。核心目標是「5 分鐘看懂一個陌生專案」，但第一印象就卡頓 + 視覺混亂，根本做不到 |
| 頻率 | 每次使用都會遇到 — 卡頓影響所有操作，3D 空間感缺失影響每次 3D 瀏覽 |
| 嚴重度 | 極高 — 這是核心體驗問題，不是功能缺失。「視覺衝擊力就是產品本身」，卡頓和混亂直接摧毀產品核心價值。開源前必須解決 |
| 現有替代方案 | 無。使用者無法自行解決卡頓或 3D 空間感問題 |

**結論**: [x] 問題真實存在且值得解決

## Q2: 目標用戶是誰？（User Definition）

| 檢查項 | 說明 |
|--------|------|
| 用戶角色 | 所有使用者。效能和視覺是基礎體驗，影響每一個人 |
| 使用頻率 | 每次使用 |
| 技術水平 | 零門檻 — 效能好 = 自然流暢，3D 空間感好 = 自然看懂 |
| 關鍵場景 | ① 第一次打開 CodeAtlas → 3D 模式 → 看到節點在空間中分佈，有軸線和網格 → 「哇，這像星圖」② 切換視圖、觸發影響分析 → 流暢無卡頓 → 「好順」 |

**結論**: [x] 用戶輪廓清晰

---

## Q3: 解法合理嗎？（Solution Fitness）

| 檢查項 | 說明 |
|--------|------|
| 複雜度 vs 價值 | **效能優化**：ViewState 拆分（避免全局 re-render）+ useMemo/useCallback 優化 + graph-adapter 快取 + 3D 渲染減負。複雜度中等，價值極高。**3D 空間參考系**：Three.js GridHelper + AxesHelper + 標籤，複雜度低，視覺提升大 |
| 有沒有更簡單的做法 | 效能：可先做 profiling 找到最大瓶頸，針對性優化而非全面重構。3D 空間：Three.js 內建 GridHelper / AxesHelper，幾行程式碼就有基礎效果 |
| 技術可行性 | ✅ 效能優化是標準 React + Three.js 優化手法，不需新技術。3D 空間參考系是 Three.js 基礎功能 |
| 邊界情況 | ① 效能優化可能改到 ViewStateContext 結構，影響所有 Sprint 的功能 → 需完整回歸 ② 3D 網格在深色主題下要夠淡不搶視覺 → 需調色 ③ 網格/軸線在 2D 模式不適用 → 只限 3D |

**結論**: [x] 解法合理

## Q4: 範圍明確嗎？（Scope Clarity）

| 檢查項 | 說明 |
|--------|------|
| 必做（Must Have） | ① 效能 profiling + 瓶頸定位 ② ViewState 優化（拆分 context 或 selector 機制，避免全局 re-render）③ graph-adapter 計算快取（useMemo + 依賴精確化）④ 3D 渲染效能（減少 useEffect、光效分級）⑤ 3D XYZ 軸線 + 背景網格 ⑥ 3D 軸標示（旋轉時始終可見）⑦ 全面回歸測試 |
| 應做（Should Have） | Sprint 9 T13 P1 收尾（AI 設定 UI + Onboarding）、節點佈局優化（避免擠成一團 — 調整力導向參數） |
| 不做（Out of Scope） | 多語言 Python/Java（順延 Sprint 11）、Web Worker 分段計算（Phase 4 如仍需要）、自訂主題 |
| 驗收標準 | 操作流暢無明顯卡頓 + 3D 模式有 XYZ 軸 + 背景網格 + 旋轉時能判斷方位 |
| 依賴 | Sprint 1-9 所有功能（需完整回歸） |

**結論**: [x] 範圍清晰可執行
